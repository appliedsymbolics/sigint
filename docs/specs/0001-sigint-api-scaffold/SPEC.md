# SPEC

## Summary

Create the initial `sigint` Go API project as a generic events ingest service with GORM-backed persistence, a retained SQLite option, generated OpenAPI documentation, and `sigint` / `SIGINT` naming.

## Problem

`sigint` had repository instruction scaffolding but no runtime implementation. The project needs a production-ready starting point for a generic event collection API with practical primitives: CLI operations, server lifecycle, health endpoints, Docker, Compose, local examples, generated API documentation, tests, and clear docs.

## Goals

1. Establish a generic events ingest service structure and conventions as the starting point for `sigint`.
2. Provide a Go module, `cmd/sigint` entrypoint, Cobra CLI, Gin HTTP API, config loader, runtime wiring, GORM persistence, storage adapters, examples, tests, Makefile, Dockerfile, and standalone `docker-compose.yaml`.
3. Use GORM for Postgres while retaining SQLite as the default local option.
4. Use generated OpenAPI docs with Swaggo annotations, generated `docs/docs.go`, `docs/swagger.json`, and `docs/swagger.yaml`, and Swagger UI served by the running API.
5. Keep generic event collection behavior: single and batch ingest, lookup, internal replay, health/readiness, debug mode, canonical hashing, duplicate success, and hash-conflict handling.
6. Keep `docker-compose.yaml` standalone.
7. Update README with practical operator docs.
8. Include the existing uncommitted scaffold/docs work in the new GH-1 delivery lane.

## Non-Goals

1. Do not add domain-specific business rules.
2. Do not make any sibling repository or external compose stack required for local sigint development.
3. Do not introduce queues, workflow engines, outbox delivery, or domain-specific projections.
4. Do not make Postgres, Docker, LocalStack, or cloud credentials mandatory for the default local workflow.
5. Do not introduce hand-written SQL persistence as the primary ledger path; this feature intentionally uses GORM.

## Acceptance Criteria

1. `go.mod` defines `github.com/appliedsymbolics/sigint` and includes the required runtime and docs dependencies.
2. `cmd/sigint` builds a CLI with server, db, ingest, events, retention, and diagnostics commands.
3. The HTTP API exposes `/`, `/llms.txt`, `/v1/docs/swagger/*any`, `/openapi.json`, `/healthz`, `/readyz`, `/v1/events`, `/v1/events:batch`, `/v1/events/:event_id`, and `/internal/v1/events/replay`.
4. Debug routes are registered only when `SIGINT_ENV=debug`.
5. Persistence supports `sqlite` and `postgres` through GORM and AutoMigrate.
6. Local storage supports filesystem raw archive; production-like storage supports S3.
7. The default local config uses SQLite plus filesystem storage.
8. `Dockerfile` builds a distroless runtime image.
9. `docker-compose.yaml` is standalone and includes the app plus optional Postgres and LocalStack profiles.
10. `Makefile` includes build, docs, fmt, tidy, test, vet, run, dev, db-init, post-event, smoke, check, clean, and Compose targets.
11. OpenAPI docs are generated by `make docs`.
12. README includes practical usage sections for a generic events ingest service.
13. `docs/PROJECT_PROGRESS_SUMMARY.md` reflects this feature as the highest completed artifact.
