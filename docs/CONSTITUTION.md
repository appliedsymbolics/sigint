# CONSTITUTION

## Purpose

This document is the canonical project contract for `sigint`. It defines the durable rules for development, architecture, implementation, documentation, validation, and delivery. Every feature spec, ad hoc change, review, and automation prompt must preserve these invariants unless the constitution is intentionally updated first.

`sigint` is a generic events ingest service. The project should evolve into a small, auditable, production-ready event collection system for event-oriented workflows. The long-term direction is reliable ingestion, normalization, validation, provenance, and replay of event envelopes with minimal operational surprise.

## Authority

Authority order:

1. Safety and permission constraints.
2. Current user request.
3. `docs/CONSTITUTION.md`.
4. Current feature `docs/specs/<feature>/SPEC.md`.
5. Legacy `PLAN.md` and `TASKS.md` only when the user explicitly chooses a legacy staged flow.
6. Legacy `BRAINSTORM.md`.
7. Existing repo conventions.

Repo-local docs under `docs/` are the source of truth. Top-level instruction files such as `AGENTS.md`, `CLAUDE.md`, and `.github/copilot-instructions.md` are routing tables, not full manuals. They must stay aligned with `docs/agents/*` and this constitution.

## Current Baseline

- The confirmed product identity is `sigint`, a generic events ingest service.
- The module path is `github.com/appliedsymbolics/sigint`.
- The Go baseline is Go `1.25+`.
- The executable entrypoint is `cmd/sigint`.
- The runtime is a Go API and CLI for durable event ingestion.
- The HTTP API uses Gin and exposes `/`, `/llms.txt`, `/openapi.json`, `/v1/docs/swagger/*any`, `/healthz`, `/readyz`, `/v1/events`, `/v1/events:batch`, `/v1/events/:event_id`, and `/internal/v1/events/replay`.
- The CLI uses Cobra and owns server lifecycle, database initialization, file ingest, event lookup/replay, retention, and diagnostics commands.
- The ledger uses GORM with SQLite for default local development and PostgreSQL for production-shaped runs.
- Raw envelopes are archived through filesystem storage locally and S3-compatible storage for production-shaped runs.
- OpenAPI docs are generated by Swaggo into `docs/docs.go`, `docs/swagger.json`, and `docs/swagger.yaml`.
- Local and operator primitives include `Makefile`, `.air.toml`, `Dockerfile`, standalone `docker-compose.yaml`, and `examples/*`.
- Runtime behavior, dependencies, APIs, schemas, and test commands must remain backed by source code, feature specs, reference docs, or lockfiles in this repo.

## Principles

- Correctness over speed.
- Minimal, production-ready changes over broad rewrites.
- Explicit behavior over clever behavior.
- Root cause fixes over symptom patches.
- Simple viable design before abstraction.
- Clear naming and direct control flow over indirection.
- Evidence-backed claims over memory or guesswork.
- Documentation must match behavior before work is called complete.

Implementation should be boring, observable, and easy to operate. Prefer small modules with clear ownership, explicit error paths, deterministic behavior, and narrow public surfaces. Remove dead code, unused exports, and unnecessary configuration rather than carrying speculative flexibility.

## Product Goals

- Collect signal/event observations reliably.
- Preserve provenance, timestamps, source identity, and processing state.
- Normalize incoming signals into stable contracts without losing source context.
- Make event handling idempotent, replay-aware, and auditable.
- Keep failures explicit and recoverable.
- Provide operational evidence through logs, metrics, health checks, and validation artifacts when implementation exists.
- Keep integration boundaries clear so new sources, sinks, and transports can be added without corrupting core event semantics.

## Product Non-Goals

- Do not become a general-purpose data lake, message broker, SIEM, analytics warehouse, or workflow engine unless a future approved spec changes the scope.
- Do not introduce broad plugin frameworks before at least two concrete integrations prove the shape.
- Do not hide data loss, retries, partial failures, or schema drift behind silent best-effort behavior.
- Do not make agent instructions, generated state, or imported rulesets the product architecture.
- Do not document imagined runtime behavior as if it exists.

## Architectural Strategy

The core architecture separates durable domain behavior from transport and infrastructure concerns.

Current boundaries:

- `cmd/sigint` is the executable entrypoint and the Swaggo root annotation location.
- `internal/api` owns Gin routing, middleware, auth checks, docs routes, health, readiness, debug routes, and HTTP error mapping.
- `internal/api/controller` owns HTTP request/response translation and Swaggo operation annotations.
- `internal/cli` owns Cobra commands and operator workflows.
- `internal/config` owns YAML config loading, environment expansion, path resolution, and validation.
- `internal/events` owns envelope shape, canonical hashing, validation, replay query types, and cursor normalization.
- `internal/ingest` owns idempotency, conflict handling, raw archive writes, replay, and retention orchestration.
- `internal/ledger` owns GORM-backed SQLite/PostgreSQL persistence and AutoMigrate schema setup.
- `internal/storage` owns filesystem and S3 raw-envelope archive adapters.
- `internal/runtime` wires config, ledger, storage, and ingest service instances together.
- `internal/debugstream` owns local debug event streaming only.

Core event contracts should be append-oriented and idempotency-aware. Any feature that introduces event shape, source metadata, delivery acknowledgement, retry semantics, or persistence must document:

- Required fields.
- Optional fields.
- Schema or contract version.
- Source identity and provenance.
- Observed time and received time semantics.
- Idempotency key or duplicate-handling strategy.
- Validation errors and partial-failure behavior.
- Backward compatibility expectations.

Unknown fields, malformed payloads, duplicate events, and downstream failures must have explicit behavior. Silent dropping is not acceptable unless the spec documents why the data is intentionally discarded and how operators can observe it.

## Implementation Strategy

- Build the narrowest vertical slice that proves the contract.
- Prefer standard library and stable, well-maintained dependencies.
- Add dependencies only when they remove real risk or meaningful complexity.
- Keep dependency purpose explicit in the feature spec or reference docs.
- Avoid public APIs until a caller exists.
- Keep interfaces small and define them at the consumer boundary.
- Prefer dependency injection through constructors or explicit parameters over package-level mutable state.
- Prefer context-aware operations for I/O, long-running work, shutdown, and cancellation.
- Treat retries, timeouts, and backoff as product behavior, not incidental library defaults.
- Preserve user data and source payloads carefully; never log secrets, credentials, tokens, or sensitive event payloads by default.

Follow idiomatic Go:

- Short lowercase package names.
- Export only symbols that are required outside the package.
- Return errors explicitly and wrap them with useful context.
- Keep `context.Context` as the first parameter for operations that cross process, network, disk, or cancellation boundaries.
- Use table tests for contract-heavy behavior.
- Keep generated code clearly separated from hand-written code. Swaggo output belongs under `docs/` and must be regenerated through `make docs`.

## Code Style And Naming

- Use names from the domain: signal, event, source, observation, provenance, collector, ingestion, delivery, acknowledgement, retry, and audit.
- Avoid overloaded names such as manager, helper, utils, processor, or service when a precise role exists.
- File and package boundaries should reflect ownership, not convenience.
- Keep functions short enough to read without hidden control flow.
- Prefer explicit branching over clever maps of callbacks for core behavior.
- Prefer structured data and parsers over ad hoc string manipulation.
- Comments should explain non-obvious decisions, invariants, or tradeoffs; they should not narrate obvious code.
- Source files should stay around 300 lines when splitting improves clarity and ownership.

- `docs/specs/<feature>/SPEC.md` is the v2 feature source of truth.

## Feature Process

All work must be classified before acting.

Use spec-driven work for new features, substantial behavior changes, cross-component changes, or work that already has feature docs. In v2 feature work, `SPEC.md` owns thesis, context, clarifications, requirements, assumptions, acceptance criteria, implementation plan, task checklist, validation map, reflection notes, documentation updates, delivery decision, and evidence.

Before implementing a feature:

- Clarify until unresolved assumptions are zero.
- Make acceptance criteria binary-verifiable.
- Map tasks to acceptance criteria.
- Map validation one-to-one with the behavior being changed.
- Resolve delivery intent.
- Load only the feature's canonical front matter skills, references, relationships, and rules needed for the immediate decision.

Use ad hoc work for contained bug fixes, reviews, dependency updates, config changes, and small refinements. Ad hoc work still requires inspecting relevant files, following existing patterns, making minimal changes, and verifying with the smallest relevant checks.

If implementation changes behavior, requirements, validation, or delivery expectations, update the canonical docs first or in the same change. If documented behavior diverges from reality, correct the docs before treating the work as complete.

## Progress Summary Invariant

`docs/PROJECT_PROGRESS_SUMMARY.md` must reflect the highest completed artifact per feature at all times.

When a feature artifact advances, regresses, is superseded, or is intentionally skipped, update or regenerate the progress summary in the same change. If no features exist, the summary may have no feature rows, but it must not imply progress that is not backed by canonical feature artifacts.

Do not edit generated progress state incidentally. If progress-summary churn appears outside the requested scope, inspect the cause and keep only changes required by the current task.

## Validation And Evidence

- Never claim tests passed unless they ran.
- Never claim files were inspected unless they were inspected.
- Never guess file contents, APIs, dependency behavior, or runtime behavior.
- If validation cannot run, state why.
- Fix relevant lint, format, and test failures before calling work complete.
- For feature work, keep validation evidence in `SPEC.md`.
- For repo-wide stable testing norms, update `docs/references/testing.md`.
- For runtime services, validation must cover startup, shutdown, failure paths, and contract behavior introduced by the change.

Validation should be as small as possible while still proving the changed behavior. Prefer focused checks first, then broader suites when shared contracts or cross-module behavior changed.

## Git And Delivery

This is a Kit-managed project. Issue, branch, staging, commit, push, and pull request actions are mutation boundaries.

Before any GitHub delivery mutation, load the repo-local delivery rules, run recon, resolve the delivery contract, and stop if any required field is unknown. Repo-local Kit rules outrank global agent or plugin defaults.

Non-negotiable delivery rules:

- Work in the existing project directory.
- Do not create or use git worktrees for agent work.
- Do not commit to protected or assumed-protected base branches.
- Do not stage with `git add .` or `git add -A`.
- Do not run `git add` or `git commit` without explicit user approval.
- Do not force-push, rebase, reset hard, delete branches, alter repository settings, or bypass review.
- The human user must be the git author and committer.
- Do not add agent attribution trailers.
- Use the repo-defined `GH-123` branch, commit, and PR conventions when PR delivery is requested.
- Create ready PRs by default only when the delivery rules and user request authorize PR creation.
- Report GitHub checks exactly; do not imply CI success from pending, missing, skipped, or failed checks.

Standalone documentation, planning, read-only, review-only, and exploratory work must not be forced into issue, branch, commit, push, or PR workflow unless the user explicitly asks.

## Safety And Security

- Keep changes minimal and reversible.
- Never stage secrets, `.env` files, tokens, private keys, or machine-local config.
- Do not read or expose local secret files unless the user explicitly asks and the task requires it.
- Fail closed on ambiguous branch protection, identity, authentication, or delivery state.
- Stop on failures that affect correctness, validation, or delivery. Report the exact error and current state instead of mutating around the failure.
- Sensitive event payloads must be redacted or excluded from logs by default once runtime code exists.

## Web And API Surface

This repository exposes an HTTP API and must provide a maintained root `/llms.txt` endpoint according to `docs/references/rules/llms-txt.md`.

`/llms.txt` must be concise, Markdown-formatted, current with public routes and docs, and free of secrets or private operational details. It complements authorization, OpenAPI, `robots.txt`, `sitemap.xml`, and human documentation; it does not replace them.

API route rules:

- Keep `/healthz` lightweight.
- Keep `/readyz` dependency-aware.
- Keep `/openapi.json` generated from Swaggo output.
- Keep `/v1/docs/swagger/*any` aligned with generated OpenAPI docs.
- Keep protected routes optional by config: no token means no bearer check, configured token means exact bearer-token auth.
- Keep debug routes behind `SIGINT_ENV=debug`; never make debug routes part of the default public surface.

Configuration rules:

- `SIGINT_CONFIG` is the environment entrypoint for loading YAML config.
- Config values may reference environment variables, but missing secret variables must fail closed.
- Default local development uses SQLite plus filesystem storage.
- Production-shaped configs use PostgreSQL plus S3-compatible storage.
- SQLite and filesystem paths are resolved relative to the config file directory.

## Change Classification

### Spec-Driven

Use for new features, substantial architectural or behavioral changes, cross-component changes, or work that already has feature docs.

Workflow: `kit spec <feature>` creates or updates `docs/specs/<feature>/SPEC.md` through clarify, ready, implement, validate, reflect, and deliver phases.

Legacy staged artifacts (`BRAINSTORM.md`, legacy `SPEC.md`, `PLAN.md`, `TASKS.md`) are historical unless explicitly chosen.

### Ad Hoc

Use for bug fixes, security reviews, refactors, dependency updates, config changes, and small refinements.

Workflow: understand, implement, verify.

Update only practical docs unless existing feature docs or repo-wide references are affected.

### Ad Hoc With Existing Specs

If a change touches code or behavior governed by existing feature docs, default to updating those docs.

Skip spec updates only for purely mechanical changes such as formatting, typo fixes, or dependency bumps that do not change behavior or validation expectations.

## Definitions

- Signal: A domain observation or event worth collecting, validating, and preserving.
- Event: A concrete signal instance received from a source.
- Source: The producer, integration, system, file, user action, or transport that supplied an event.
- Provenance: Evidence that explains where an event came from, when it was observed, when it was received, and how it was transformed.
- Ingress: The boundary where external input enters the collector.
- Normalization: Conversion from source-specific input into an internal contract.
- Delivery: Persisting, forwarding, or acknowledging an event after validation.
- Idempotency: Handling duplicate inputs without duplicating durable effects.
- Feature artifact: The canonical `SPEC.md` or legacy staged document representing a feature's highest completed development phase.
- Highest completed artifact: The most advanced verified phase or document state for a feature, as reflected in `docs/PROJECT_PROGRESS_SUMMARY.md`.
